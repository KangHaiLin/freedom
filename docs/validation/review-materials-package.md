## 评审工作流程

### 阶段1：评审准备（第1周）
```
1. 材料分发 → 2. 个人预审 → 3. 问题标注 → 4. 反馈收集
```

**具体任务**：
1. **材料分发**：将本材料包发送给所有评审团队成员
2. **个人预审**：各成员根据专业领域预审相关部分
3. **问题标注**：使用文档评注工具或问题跟踪模板记录问题
4. **反馈收集**：在启动会议前收集初步反馈

### 阶段2：会议评审（第2-3周）
```
1. 启动会议 → 2. 专题评审 → 3. 综合评审 → 4. 结论会议
```

**具体任务**：
1. **启动会议**：明确评审目标、方法、时间安排
2. **专题评审**：按业务、技术、合规等专题分组评审
3. **综合评审**：解决跨领域问题，达成共识
4. **结论会议**：形成最终评审结论

### 阶段3：问题解决（贯穿始终）
```
1. 问题记录 → 2. 问题分析 → 3. 方案制定 → 4. 方案实施 → 5. 效果验证
```

**具体任务**：
1. **问题记录**：使用问题跟踪模板记录所有问题
2. **问题分析**：分析问题原因和影响
3. **方案制定**：制定解决方案和改进措施
4. **方案实施**：实施解决方案
5. **效果验证**：验证解决方案的有效性

## 评审方法指南

### 1. 基于检查清单的评审

**使用工具**：`requirement-validation-checklist.md`

**步骤**：
1. 每位评审员根据专业领域选择相关检查项
2. 逐项检查并记录结果
3. 汇总检查结果，识别薄弱环节
4. 针对未通过项制定改进措施

**检查重点**：
- 业务代表：完整性、一致性、业务可行性
- 技术代表：技术可行性、可测试性、可维护性
- 合规代表：合规性、安全性、审计要求
- 运维代表：可操作性、可靠性、可移植性

### 2. 基于场景的评审

**使用工具**：`../requirements/use-cases.md`

**步骤**：
1. 选择关键用户场景（如量化研究员开发策略）
2. 追踪场景中涉及的每个需求
3. 验证需求是否支持场景的完整流程
4. 识别场景中缺失或不完整的需求

**场景选择建议**：
- 量化研究员开发新策略（UC-01）
- 交易员执行算法交易（UC-02）
- 风险经理监控系统风险（UC-03）
- 系统管理员日常运维（UC-04）
- 合规官进行合规检查（UC-05）

### 3. 基于原型的评审

**适用情况**：复杂功能或创新功能

**步骤**：
1. 创建关键功能的简单原型
2. 演示原型验证需求可行性
3. 收集用户反馈和改进建议
4. 根据反馈更新需求

**建议原型方向**：
- 数据API调用示例
- 交易界面布局原型
- 风险监控仪表板设计

## 评审工具使用指南

### 1. 文档评注工具

**推荐工具**：
- **在线协作工具**：Confluence评论、Google Docs建议模式
- **专业评审工具**：Review Board、Gerrit
- **简单替代方案**：Word/PDF批注功能

**使用建议**：
1. 为每个评注添加标签（如：[业务]、[技术]、[合规]）
2. 明确标注问题位置（章节号、行号）
3. 提供具体建议而不仅仅是批评
4. 使用统一的问题严重性标识（严重、重要、一般）

### 2. 会议管理工具

**推荐工具**：
- **会议工具**：Zoom、Teams、腾讯会议（支持录制和转录）
- **白板工具**：Miro、Mural、Jamboard
- **投票工具**：Poll Everywhere、Slido

**使用建议**：
1. 会前发布议程和预审材料
2. 会议中记录关键决策和行动项
3. 会后24小时内发布会议纪要
4. 使用投票工具解决争议问题

### 3. 问题跟踪工具

**推荐工具**：
- **专业工具**：JIRA、Bugzilla、Redmine
- **简单工具**：Excel模板、Google Sheets
- **协作工具**：Trello、Asana

**使用建议**：
1. 为每个问题分配唯一ID
2. 明确问题责任人、解决期限
3. 定期更新问题状态
4. 问题解决后验证关闭

## 评审产出物要求

### 1. 必须产出物

| 产出物 | 负责人 | 提交时间 | 质量要求 |
|--------|--------|----------|----------|
| **个人预审反馈** | 各评审员 | 启动会议前 | 至少覆盖专业领域相关章节 |
| **会议纪要** | 会议记录员 | 会后24小时内 | 包含决策、行动项、待解决问题 |
| **问题清单** | 评审协调员 | 每次专题评审后 | 问题描述清晰、分类准确、责任明确 |
| **评审报告** | 评审委员会 | 结论会议前 | 全面、客观、有建设性 |

### 2. 质量要求

**完整性要求**：
- [ ] 所有评审参与者的反馈都已收集和考虑
- [ ] 所有识别的问题都已记录和跟踪
- [ ] 所有决策都有记录和依据
- [ ] 所有产出物都已完成并归档

**一致性要求**：
- [ ] 评审结论基于客观事实和标准
- [ ] 问题处理遵循既定流程
- [ ] 决策过程透明、公正
- [ ] 产出物格式和内容一致

**实用性要求**：
- [ ] 评审结果为后续工作提供清晰指导
- [ ] 问题解决方案具体可行
- [ ] 改进建议有明确实施路径
- [ ] 签署文件具有法律效力

## 常见问题与解决方案

### Q1：评审意见不一致怎么办？
**解决方案**：
1. 组织专题讨论会深入分析
2. 引入第三方专家意见
3. 进行原型验证或概念验证
4. 采用投票机制集体决策

### Q2：评审进度延误怎么办？
**解决方案**：
1. 调整会议安排，增加会议频率
2. 分组并行评审提高效率
3. 简化评审流程，聚焦关键问题
4. 延长评审周期，确保质量

### Q3：技术可行性难以评估怎么办？
**解决方案**：
1. 进行概念验证（Proof of Concept）
2. 咨询外部技术专家
3. 参考类似项目经验
4. 采用风险评估方法

### Q4：合规要求不明确怎么办？
**解决方案**：
1. 咨询合规部门或法律顾问
2. 研究监管机构官方文件
3. 参考行业最佳实践
4. 采用最严格标准确保合规

**业务需求检查点**：
1. 量化研究员能否高效获取所需数据？

3. 安全机制是否足够保护敏感数据？

**合规需求检查点**：
1. 是否满足T+1交易制度要求？
2. 是否支持涨跌停限制检查？
3. 是否满足数据保存6年要求？
4. 是否支持监管报告生成？
