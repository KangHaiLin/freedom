### 2.3 测试数据准备

#### 数据模型定义（A股Tick数据）
```yaml
Tick数据模型：
- 测量（measurement）： tick_data
- 标签（tags）：
  - exchange: [SSE, SZSE, BSE]（交易所）
  - symbol: [SH600000, SZ000001, ...]（股票代码）
  - board: [main, STAR, ChiNext]（板块）
  - is_active: [true, false]（是否活跃股票）
- 字段（fields）：
  - price: float（最新价）
  - volume: integer（成交量）
  - amount: float（成交额）
  - bid_price_1..5: float（买一至买五价）
  - bid_volume_1..5: integer（买一至买五量）
  - ask_price_1..5: float（卖一至卖五价）
  - ask_volume_1..5: integer（卖一至卖五量）
  - open: float（开盘价）
  - high: float（最高价）
  - low: float（最低价）
  - pre_close: float（昨收价）
  - limit_up: float（涨停价）
  - limit_down: float（跌停价）
- 时间戳： 纳秒精度时间戳
- 预计每条记录大小： 约200-300字节
```

#### 数据规模定义
```
测试数据规模：
1. 基础测试数据：
   - 股票数量： 1000只（活跃股票）
   - 数据频率： 10次/秒/股票（平均）
   - 测试时长： 30分钟（持续写入）
   - 总数据点： 1000只 × 10次/秒 × 1800秒 = 18,000,000个数据点
   - 数据量： 约3.6-5.4GB

2. 峰值测试数据：
   - 股票数量： 1000只
   - 数据频率： 50次/秒/股票（峰值）
   - 测试时长： 10分钟（峰值压力）
   - 总数据点： 1000只 × 50次/秒 × 600秒 = 30,000,000个数据点
   - 数据量： 约6-9GB

3. 稳定性测试数据：
   - 股票数量： 5000只（全市场）
   - 数据频率： 5次/秒/股票（全天平均）
   - 测试时长： 8小时（交易时段）
   - 总数据点： 5000只 × 5次/秒 × 28800秒 = 720,000,000个数据点
   - 数据量： 约144-216GB
```

#### 数据质量要求
- [☑] 数据完整性：时间序列连续，无跳秒
- [☑] 数据准确性：价格和成交量符合市场特征
- [☑] 数据一致性：跨字段数据逻辑一致（如价格在涨跌停范围内）
- [☑] 数据时效性：时间戳精确到毫秒，符合实际交易时间

## 3. 测试场景设计

### 3.1 场景1：基础写入性能测试（平均负载）

#### 场景描述
验证数据库在平均负载（10K/秒）下的写入性能，评估基础目标的可行性。

#### 测试条件
| 参数 | 数值 | 说明 |
|------|------|------|
| **写入速率** | 10,000次/秒 | 目标基础性能指标 |
| **并发客户端** | 10个 | 模拟多数据源写入 |
| **数据模型** | 完整Tick数据（约250字节/记录） | 包含买卖盘信息 |
| **测试时长** | 30分钟 | 稳定状态测试 |
| **数据持久化** | 同步写入（ack确认） | 确保数据持久性 |
| **批处理大小** | 100-1000条/批 | 优化批处理大小测试 |

#### 性能指标
| 指标 | 目标值 | 测量方法 | 重要性 |
|------|--------|----------|--------|
| **写入吞吐量** | ≥10,000次/秒 | 成功写入的数据点计数 | 核心指标 |
| **P50写入延迟** | ≤50ms | 50%请求的延迟 | 用户体验 |
| **P95写入延迟** | ≤100ms | 95%请求的延迟 | 性能稳定性 |
| **P99写入延迟** | ≤200ms | 99%请求的延迟 | 尾部延迟 |
| **写入成功率** | ≥99.9% | 成功写入的比例 | 数据可靠性 |
| **CPU使用率** | ≤70% | 数据库进程CPU使用 | 资源效率 |
| **内存使用** | 稳定增长 | 内存增长趋势 | 内存管理 |

### 3.2 场景2：峰值写入性能测试（压力测试）

#### 场景描述
验证数据库在峰值负载（50K/秒）下的性能和稳定性，模拟开盘、收盘等高并发场景。

#### 测试条件
| 参数 | 数值 | 说明 |
|------|------|------|
| **写入速率** | 50,000次/秒 | 目标峰值性能指标 |
| **并发客户端** | 50个 | 高并发压力 |
| **数据模型** | 简化Tick数据（约150字节/记录） | 核心字段，减少网络开销 |
| **测试时长** | 10分钟 | 短期峰值压力 |
| **预热阶段** | 5分钟（10K/秒） | 逐步增加到峰值 |
| **批处理优化** | 动态批处理大小 | 根据延迟调整 |

#### 性能指标
| 指标 | 目标值 | 测量方法 | 可接受范围 |
|------|--------|----------|------------|
| **峰值吞吐量** | ≥50,000次/秒 | 峰值期间的平均吞吐量 | ≥40,000次/秒 |
| **P95写入延迟** | ≤500ms | 峰值期间P95延迟 | ≤1000ms |
| **P99写入延迟** | ≤1000ms | 峰值期间P99延迟 | ≤2000ms |
| **错误率** | ≤1% | 写入失败的比例 | ≤5%（短期峰值） |
| **队列积压** | ≤1000条 | 等待写入的队列长度 | ≤5000条 |
| **资源峰值** | CPU≤90% | 峰值期间资源使用 | 短期可接受 |

### 3.3 场景3：混合读写性能测试（生产场景）

#### 场景描述
验证数据库在同时处理写入和查询负载下的性能，模拟真实生产环境。

#### 测试条件
| 参数 | 数值 | 说明 |
|------|------|------|
| **写入负载** | 8,000次/秒 | 80%的写入负载 |
| **查询负载** | 100次/秒 | 20%的查询负载 |
| **查询类型** | 多种查询混合：<br>1. 单股票近期数据<br>2. 多股票聚合查询<br>3. 时间范围查询<br>4. 条件过滤查询 | 典型查询模式 |
| **并发用户** | 20个（15写+5读） | 混合工作负载 |
| **测试时长** | 60分钟 | 混合负载稳定性 |
| **数据热度** | 热数据（最近1小时） | 高频查询数据 |

#### 性能指标
| 指标 | 目标值 | 测量方法 |
|------|--------|----------|
| **写入性能影响** | ≤20%性能下降 | 与纯写入场景对比 |
| **查询P95延迟** | ≤200ms | 查询响应时间 |
| **查询成功率** | ≥99% | 查询成功比例 |
| **资源竞争** | 无明显资源竞争 | CPU、IO、锁竞争监测 |
| **缓存命中率** | ≥80%（热数据） | 查询缓存效果 |

### 3.4 场景4：长时间稳定性测试（耐力测试）

#### 场景描述
验证数据库在长时间持续写入下的稳定性和性能一致性。

#### 测试条件
| 参数 | 数值 | 说明 |
|------|------|------|
| **写入速率** | 15,000次/秒 | 高于平均，低于峰值 |
| **测试时长** | 8小时 | 模拟全天交易时段 |
| **数据保留** | 全部保留 | 测试数据膨胀影响 |
| **监控频率** | 每5分钟 | 性能数据采集 |
| **维护窗口** | 无 | 持续运行，无维护 |
| **故障注入** | 无（稳定性测试） | 纯稳定性验证 |

#### 性能指标
| 指标 | 目标值 | 测量方法 |
|------|--------|----------|
| **性能衰减** | ≤10% | 结束时的性能下降 |
| **内存泄漏** | 无持续增长 | 内存使用趋势分析 |
| **磁盘空间增长** | 线性增长 | 数据文件增长符合预期 |
| **系统稳定性** | 无崩溃、无重启 | 服务可用性监测 |
| **数据一致性** | 100%一致 | 定期数据校验 |

### 3.5 场景5：扩展性测试（集群和分片）

#### 场景描述
验证数据库水平扩展能力，评估未来数据增长的可扩展性。

#### 测试条件
| 参数 | 数值 | 说明 |
|------|------|------|
| **节点数量** | 1, 2, 3个节点 | 扩展性对比 |
| **数据分片策略** | 按股票代码哈希分片 | 均匀分布负载 |
| **写入负载** | 20,000次/秒 | 固定负载测试扩展性 |
| **测试配置** | 1. 单节点基准<br>2. 双节点集群<br>3. 三节点集群 | 扩展效率对比 |
| **故障转移** | 模拟节点故障 | 集群容错能力 |

#### 性能指标
| 指标 | 目标值 | 测量方法 |
|------|--------|----------|
| **扩展效率** | ≥70% | 性能提升与资源增加比例 |
| **数据分布均衡** | 标准差≤20% | 各节点负载均衡度 |
| **故障转移时间** | ≤30秒 | 节点故障恢复时间 |
| **数据一致性** | 最终一致/强一致 | 集群数据一致性 |
| **管理复杂度** | 可接受 | 集群运维复杂度评估 |

```python
# 存储性能分析框架
import pandas as pd
import numpy as np
from scipy import stats

class StoragePerformanceAnalyzer:
    def __init__(self, test_data):
        self.data = test_data

    def analyze_scenario(self, scenario_name, db_technology):
        """分析特定场景和技术的性能数据"""
        scenario_data = self.data[
            (self.data['scenario_id'] == scenario_name) &
            (self.data['db_technology'] == db_technology)
        ]

        analysis = {
            'scenario': scenario_name,
            'technology': db_technology,
            'samples': len(scenario_data),
            'throughput_stats': self._calc_stats(scenario_data, 'throughput'),
            'latency_stats': self._calc_stats(scenario_data, 'p95_latency'),
            'error_stats': self._calc_stats(scenario_data, 'error_rate'),
            'resource_usage': self._analyze_resources(scenario_data)
        }

        # 目标达成评估
        analysis['target_assessment'] = self._assess_targets(analysis)

        return analysis

    def compare_technologies(self, scenario_name):
        """比较不同技术在相同场景下的表现"""
        technologies = self.data['db_technology'].unique()
        comparison = {}

        for tech in technologies:
            comparison[tech] = self.analyze_scenario(scenario_name, tech)

        return comparison
```

### 6.3 应急计划

#### 问题分类和响应
- **P0紧急问题**：测试环境完全不可用
  - **响应流程**：立即切换到备份环境，通知所有相关人员
  - **升级机制**：15分钟内通知项目负责人，30分钟内制定恢复计划
  - **记录要求**：详细记录故障现象、影响范围、恢复步骤

- **P1严重问题**：关键测试场景无法执行或结果无效
  - **响应流程**：暂停相关测试，分析原因，制定修复方案
  - **时间窗口**：2小时内解决问题或调整测试计划
  - **沟通要求**：每小时更新进展，影响评估报告

- **P2一般问题**：非关键问题，测试可继续
  - **响应流程**：记录问题，评估影响，计划后续修复
  - **时间窗口**：4小时内制定处理计划
  - **报告要求**：每日测试报告中汇总

- **P3轻微问题**：不影响测试有效性的问题
  - **响应流程**：记录问题，测试完成后处理
  - **时间窗口**：测试完成后统一处理
  - **文档要求**：测试报告附录中记录

#### 测试计划调整机制
- **调整条件**：环境问题、技术问题、时间问题导致原计划不可行
- **调整流程**：提出调整方案→影响评估→相关方批准→更新测试计划
- **调整原则**：优先保证核心测试场景，适当缩减次要场景
- **文档更新**：所有调整必须记录，最终报告说明调整原因和影响
