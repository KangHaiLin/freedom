# 需求评审参与指南

## 概述

本指南为A股量化交易软件SRS需求评审的参与者提供行为规范、职责要求和参与指导。所有评审参与者必须遵守本指南，确保评审过程高效、专业、富有建设性。

## 评审参与者角色与职责

### 1. 评审主席
**核心职责**：
- 总体协调评审过程，确保按计划进行
- 主持关键会议，引导讨论方向
- 决策争议问题，确保评审结论公正
- 批准最终评审报告和签署文件

**行为要求**：
- 保持中立立场，不偏袒任何一方
- 确保所有参与者有机会表达意见
- 控制会议节奏，避免无谓争论
- 及时做出决策，避免拖延

### 2. 评审委员
**核心职责**：
- 根据专业领域深入评审相关需求
- 提供专业意见和建议
- 识别需求问题、风险和不足
- 参与讨论和决策过程

**行为要求**：
- 会前充分准备，完成预审工作
- 基于事实和专业知识提出意见
- 尊重他人观点，建设性讨论
- 按时提交评审反馈和报告

### 3. 专题组长
**核心职责**：
- 组织专题评审会议，制定评审计划
- 汇总专题评审结果，编写专题报告
- 协调组内成员工作，确保评审质量
- 代表专题组参与综合讨论

**行为要求**：
- 精通专题领域知识，能够指导组员
- 有效组织会议，确保讨论聚焦
- 客观汇总意见，不加入个人偏见
- 按时完成专题报告

### 4. 记录员
**核心职责**：
- 准确记录会议讨论和决策
- 整理会议纪要，及时分发
- 维护问题跟踪表，更新状态
- 归档评审文档和产出物

**行为要求**：
- 记录准确完整，不加入个人理解
- 及时整理和分发会议纪要
- 保密评审内容，不擅自传播
- 细心整理文档，确保完整性

## 评审行为准则

### 1. 专业态度
- **尊重专业**：基于专业知识和经验提出意见
- **客观公正**：不带个人偏见，以事实为依据
- **建设性批评**：指出问题的同时提供改进建议
- **持续学习**：保持开放心态，学习他人经验

### 2. 会议行为
- **准时参会**：提前5分钟进入会议
- **充分准备**：会前阅读材料，准备问题
- **积极发言**：主动参与讨论，分享观点
- **有效倾听**：认真听取他人意见，不随意打断
- **聚焦议题**：围绕会议议题讨论，不偏离主题

### 3. 沟通规范
- **清晰表达**：观点明确，逻辑清晰
- **专业术语**：正确使用专业术语，必要时解释
- **尊重差异**：尊重不同观点，理性讨论
- **书面确认**：重要决策和行动项书面确认

### 4. 文档处理
- **保密要求**：所有评审材料严格保密
- **规范记录**：按照模板要求记录问题和建议
- **及时提交**：按时提交评审反馈和报告
- **版本控制**：使用正确版本的文件

## 评审工作方法指南

### 1. 个人预审方法
**时间要求**：会前至少预留4小时进行预审

**工作步骤**：
1. **快速浏览**：通读分配章节，了解整体内容
2. **深入阅读**：逐段仔细阅读，理解需求细节
3. **检查清单**：使用检查清单逐项检查
4. **问题标注**：在文档中标注问题和建议
5. **反馈整理**：整理问题清单和修改建议

**检查重点**：
- 需求描述是否清晰、无歧义？
- 验收标准是否具体、可测量？
- 需求之间是否有矛盾或冲突？
- 技术实现是否可行？
- 业务价值是否明确？

### 2. 专题评审方法
**准备工作**：
- 提前收集组员预审问题
- 准备专题检查清单
- 安排会议议程和时间

**会议流程**：
1. **问题汇总**：分享预审发现的问题
2. **重点讨论**：深入讨论关键问题
3. **评估分析**：评估问题影响和风险
4. **建议制定**：制定改进建议和解决方案
5. **结论形成**：形成专题评审结论

**注意事项**：
- 安排专人记录讨论要点
- 控制讨论时间，避免偏离主题
- 确保所有参与者都有发言机会
- 及时总结讨论要点和结论

### 3. 问题记录规范
**问题描述要求**：
- **具体**：明确指出问题位置（章节、行号）
- **清晰**：问题描述清楚，易于理解
- **客观**：基于事实描述，不加入主观判断
- **建设性**：提供改进建议或解决方案

**问题分类**：
- **严重问题**：影响系统核心功能或合规性
- **重要问题**：影响系统完整性或用户体验
- **一般问题**：需要改进但不影响核心功能
- **建议性意见**：改进建议，非必需修改

**记录格式**：
```
问题ID: [自动生成]
发现日期: [YYYY-MM-DD]
发现人: [姓名]
问题位置: [章节号/行号]
问题描述: [详细描述问题]
问题分类: [严重/重要/一般/建议]
影响分析: [问题的影响和风险]
建议方案: [改进建议或解决方案]
责任人: [负责解决的人员]
解决期限: [YYYY-MM-DD]
状态: [待处理/处理中/已解决/已验证]
```

### 4. 反馈提交要求
**格式要求**：
- 使用提供的模板提交反馈
- 统一文件格式（Markdown或Word）
- 包含必要的信息字段

**内容要求**：
- 覆盖分配的所有评审范围
- 提供具体的修改建议
- 区分必须修改和建议修改
- 说明修改的理由和依据

**时间要求**：
- 个人预审反馈：指定日期前提交
- 专题评审报告：评审后24小时内提交
- 会议纪要反馈：收到后48小时内确认

## 评审工具使用指南

### 1. 文档评注工具
**推荐工具**：Confluence评论、Google Docs建议模式

**使用规范**：
1. **评注格式**：
   ```
   [问题类型]: 问题描述
   建议: 修改建议
   位置: 第X章第Y节
   优先级: 高/中/低
   ```

2. **标签使用**：
   - `[业务]`: 业务相关问题
   - `[技术]`: 技术实现问题
   - `[合规]`: 合规安全问题
   - `[测试]`: 可测试性问题
   - `[文档]`: 文档质量问题

3. **评注要求**：
   - 每条评注只针对一个具体问题
   - 提供具体的修改建议
   - 标注问题严重程度
   - @相关人员引起注意

### 2. 会议工具使用
**推荐工具**：腾讯会议、Zoom、Teams

**使用规范**：
1. **会前准备**：
   - 提前测试音频视频设备
   - 下载会议材料到本地
   - 准备讨论要点和问题

2. **会议中**：
   - 开启摄像头（如无特殊要求）
   - 发言时开启麦克风，结束后静音
   - 使用举手功能申请发言
   - 共享屏幕时只分享必要内容

3. **会后**：
   - 确认会议纪要准确性
   - 完成分配的行动项
   - 反馈会议效果和建议

### 3. 协作工具使用
**推荐工具**：Confluence、Google Drive、企业微信

**使用规范**：
1. **文档管理**：
   - 使用统一目录结构
   - 遵循文件命名规范
   - 及时更新文档版本

2. **沟通协调**：
   - 使用群组进行日常沟通
   - 重要决策通过邮件确认
   - 定期同步工作进展

3. **问题跟踪**：
   - 使用统一的问题跟踪工具
   - 及时更新问题状态
   - 定期汇报问题解决进展

## 常见问题处理指南

### Q1：评审意见不一致怎么办？
**处理流程**：
1. **深入讨论**：组织专题讨论会，深入分析不同观点
2. **数据支持**：收集相关数据和事实支持各自观点
3. **专家咨询**：咨询领域专家或第三方意见
4. **风险评估**：评估不同方案的风险和影响
5. **集体决策**：采用投票机制或主席决策

**注意事项**：
- 保持专业态度，避免个人情绪
- 尊重不同观点，理性讨论
- 以项目利益为重，寻求最优方案
- 记录决策过程和依据

### Q2：发现重大问题如何处理？
**处理流程**：
1. **立即报告**：第一时间向评审主席报告
2. **评估影响**：评估问题对项目的影响和风险
3. **紧急讨论**：组织紧急会议讨论解决方案
4. **制定方案**：制定应急解决方案
5. **跟踪解决**：跟踪问题解决进展，定期汇报

**报告要求**：
- 详细描述问题现象和影响
- 提供初步分析和建议
- 建议紧急程度和处理优先级
- 预估解决时间和资源需求

### Q3：评审进度延误怎么办？
**处理措施**：
1. **原因分析**：分析延误原因，识别瓶颈
2. **调整计划**：调整评审计划和时间安排
3. **增加资源**：增加评审人员或延长工作时间
4. **优化方法**：优化评审方法，提高效率
5. **重点聚焦**：聚焦关键问题，暂缓次要问题

**预防措施**：
- 制定合理的评审计划
- 预留缓冲时间应对意外
- 定期检查评审进展
- 及时调整和优化计划

### Q4：技术可行性难以评估怎么办？
**解决方法**：
1. **概念验证**：进行小规模概念验证
2. **专家咨询**：咨询外部技术专家
3. **类似参考**：参考类似项目经验
4. **风险评估**：采用风险评估方法
5. **原型开发**：开发简单原型验证可行性
