# 使用场景

## 概述

本文档描述A股量化交易软件的典型使用场景，涵盖主要用户角色和关键业务流程。使用场景有助于理解用户需求、验证系统设计、指导测试用例开发。

## 场景分类

### 1. 策略开发场景
### 2. 交易执行场景
### 3. 风险管理场景
### 4. 系统管理场景
### 5. 数据分析场景

## 详细使用场景

### 场景1：量化研究员开发新策略

#### 场景描述
量化研究员张明需要开发一个新的Alpha策略，基于财务因子和技术指标。

#### 参与者
- 主要参与者：量化研究员（张明）
- 次要参与者：系统（数据服务、回测引擎、策略库）

#### 前置条件
1. 张明已登录系统
2. 系统中有足够的历史数据
3. Python开发环境可用

#### 主成功场景
1. **数据探索**
   - 张明使用数据API获取目标股票池的历史价格数据
   - 使用基本面数据API获取财务报表数据
   - 使用数据可视化工具分析数据分布和相关性

2. **因子开发**
   - 编写Python代码计算财务因子（PE、PB、ROE等）
   - 计算技术指标（MACD、RSI、布林带等）
   - 因子标准化和中性化处理

3. **策略构建**
   - 使用因子构建选股模型
   - 定义买卖信号和仓位管理规则
   - 设置交易成本模型（佣金、印花税、冲击成本）

4. **回测验证**
   - 配置回测参数（时间范围、初始资金、股票池）
   - 运行回测，获取绩效报告
   - 分析回测结果（收益率、夏普比率、最大回撤）

5. **策略优化**
   - 对策略参数进行网格搜索优化
   - 使用交叉验证防止过拟合
   - 选择最优参数组合

6. **策略保存**
   - 将策略代码保存到策略库
   - 添加策略说明文档
   - 设置版本标签

#### 扩展场景
- **数据不足**：系统提示数据缺失，提供数据补充流程
- **回测失败**：系统提供详细错误信息和调试工具
- **绩效不佳**：研究员调整策略逻辑，重新回测

#### 后置条件
1. 新策略已保存到策略库
2. 回测报告已生成并保存
3. 策略代码已版本控制

#### 业务规则
- 回测必须包含完整的交易成本
- 策略必须通过基本的风险检查
- 所有策略代码必须通过代码审查

#### 特殊需求
- 回测性能要求：10年日线数据回测时间<30秒
- 数据访问权限：研究员只能访问授权数据
- 策略知识产权保护

### 场景2：交易员执行算法交易

#### 场景描述
交易员李华需要为一个机构客户执行大额订单，使用VWAP算法最小化市场冲击。

#### 参与者
- 主要参与者：交易员（李华）
- 次要参与者：系统（算法引擎、订单管理、风险控制）

#### 前置条件
1. 李华已登录交易终端
2. 客户订单已录入系统
3. 风险限额检查通过
4. 交易权限验证通过

#### 主成功场景
1. **订单创建**
   - 李华选择目标股票和交易方向（买入/卖出）
   - 输入订单数量（如100,000股）
   - 选择算法类型：VWAP（成交量加权平均价格）

2. **算法配置**
   - 设置交易时间范围（如全天交易）
   - 配置参与率限制（如不超过市场成交量的20%）
   - 设置价格限制（如不高于限价）
   - 配置异常处理规则

3. **风险检查**
   - 系统实时检查客户风险限额
   - 检查集中度风险
   - 检查合规规则（涨跌停限制、T+1等）
   - 所有检查通过，订单进入待执行状态

4. **算法执行**
   - 算法引擎开始执行，根据市场成交量分配子订单
   - 实时监控市场情况，动态调整下单节奏
   - 处理成交回报，更新订单状态

5. **执行监控**
   - 李华在监控面板查看执行进度
   - 实时显示执行价格与VWAP基准的对比
   - 监控市场冲击和交易成本
   - 接收异常报警（如市场异常波动）

6. **订单完成**
   - 订单全部成交或到达截止时间
   - 系统生成执行报告
   - 计算执行绩效（实施差额、冲击成本等）
   - 订单状态更新为已完成

#### 扩展场景
- **风险检查失败**：订单被拒绝，显示具体原因
- **市场异常**：算法暂停，等待人工干预
- **部分成交**：剩余数量可选择继续执行或取消

#### 后置条件
1. 订单执行完成，生成执行报告
2. 交易记录已保存到数据库
3. 风险限额已更新
4. 客户持仓已更新

#### 业务规则
- 交易时间必须在A股交易时间内
- 单笔订单不得超过涨跌停价格
- 买入订单必须有足够资金
- 卖出订单必须有足够持仓（T+1检查）

#### 特殊需求
- 实时性要求：订单提交到券商响应<2秒
- 可靠性要求：交易时间系统可用性99.9%
- 审计要求：所有操作记录完整保存

### 场景3：风险经理监控系统风险

#### 场景描述
风险经理王芳在交易时间监控全系统风险，处理风险报警。

#### 参与者
- 主要参与者：风险经理（王芳）
- 次要参与者：系统（风险计算引擎、监控仪表盘、报警系统）

#### 前置条件
1. 王芳已登录风险管理终端
2. 系统中有实时交易数据和市场数据
3. 风险限额和规则已配置

#### 主成功场景
1. **风险仪表盘查看**
   - 王芳打开风险监控仪表盘
   - 查看整体风险概览（总市值、总风险价值VaR）
   - 查看关键风险指标趋势图

2. **限额监控**
   - 检查各账户限额使用情况
   - 查看接近限额的预警列表
   - 检查集中度风险（行业、个股）

3. **实时风险计算**
   - 系统实时计算风险指标
   - 更新风险价值（VaR）计算
   - 计算压力测试结果
   - 更新希腊字母（Delta、Gamma、Vega）

4. **风险报警处理**
   - 系统发出风险报警（如限额突破）
   - 王芳查看报警详情和原因分析
   - 评估风险状况，决定处理措施
   - 采取措施：联系交易员、调整限额、暂停交易

5. **风险报告生成**
   - 生成实时风险报告
   - 导出风险数据用于进一步分析
   - 准备监管报告所需数据

6. **风险规则调整**
   - 根据市场情况调整风险参数
   - 更新风险限额
   - 修改合规规则

#### 扩展场景
- **重大风险事件**：启动应急预案，召开紧急会议
- **系统故障**：切换备用系统，手工风险监控
- **监管检查**：提供完整风险报告和审计日志

#### 后置条件
1. 风险状况已监控和记录
2. 风险报警已处理
3. 风险报告已生成
4. 风险规则已更新（如有）

#### 业务规则
- 风险限额必须严格执行
- 风险报警必须有处理记录
- 风险报告必须每日归档
- 风险参数调整需要审批

#### 特殊需求
- 实时性要求：风险指标更新延迟<5秒
- 数据完整性：风险计算数据必须完整准确
- 审计追踪：所有风险操作必须有完整日志

### 场景4：系统管理员日常运维

#### 场景描述
系统管理员刘强进行日常系统运维，包括监控、备份、故障处理。

#### 参与者
- 主要参与者：系统管理员（刘强）
- 次要参与者：系统（监控工具、备份系统、日志系统）

#### 前置条件
1. 刘强已登录管理控制台
2. 具有系统管理员权限
3. 运维工具和脚本可用

#### 主成功场景
1. **系统健康检查**
   - 检查服务器状态（CPU、内存、磁盘、网络）
   - 检查服务进程状态
   - 检查数据库连接和性能
   - 检查消息队列状态

2. **监控和报警**
   - 查看监控仪表盘
   - 处理系统报警（如磁盘空间不足）
   - 检查业务指标异常（如交易量骤降）
   - 更新监控规则和阈值

3. **数据备份**
   - 检查昨日备份任务状态
   - 执行增量数据备份
   - 验证备份数据完整性
   - 清理过期备份文件

4. **日志分析**
   - 查看系统日志，识别潜在问题
   - 分析错误日志，定位问题原因
   - 查看安全日志，检查异常访问
   - 生成日志分析报告

5. **性能优化**
   - 分析系统性能瓶颈
   - 优化数据库查询
   - 调整系统参数
   - 清理临时文件和缓存

6. **用户支持**
   - 处理用户技术支持请求
   - 重置用户密码
   - 调整用户权限
   - 提供系统使用指导

#### 扩展场景
- **系统故障**：启动故障处理流程，恢复服务
- **安全事件**：启动安全应急响应，调查和处理
- **升级部署**：执行系统升级和部署

#### 后置条件
1. 系统状态正常
2. 备份任务完成
3. 日志已分析
4. 用户请求已处理

#### 业务规则
- 备份必须每日执行并验证
- 系统变更必须经过审批
- 安全事件必须立即报告
- 运维操作必须有记录

#### 特殊需求
- 可用性要求：交易时间系统可用性99.9%
- 安全性要求：符合网络安全等级保护要求
- 可审计性：所有运维操作完整记录

### 场景5：合规官进行合规检查

#### 场景描述
合规官赵敏进行定期合规检查，确保交易活动符合监管要求。

#### 参与者
- 主要参与者：合规官（赵敏）
- 次要参与者：系统（合规规则引擎、审计日志、报告系统）

#### 前置条件
1. 赵敏已登录合规管理界面
2. 具有合规审查权限
3. 审计日志和交易记录可用

#### 主成功场景
1. **合规规则检查**
   - 检查系统内置合规规则（T+1、涨跌停等）
   - 查看自定义合规规则执行情况
   - 分析规则触发和拦截记录

2. **交易活动审查**
   - 查询指定时间段内的所有交易
   - 筛选异常交易（如大额交易、频繁交易）
   - 审查交易对手和关联方交易
   - 检查内幕交易嫌疑

3. **客户身份验证**
   - 检查客户身份识别记录
   - 验证反洗钱（AML）检查记录
   - 审查客户风险等级评估
   - 检查客户资料完整性

4. **审计日志审查**
   - 查看系统操作审计日志
   - 检查权限变更记录
   - 分析异常访问模式
   - 验证日志完整性和不可篡改性

5. **监管报告准备**
   - 生成监管要求的定期报告
   - 准备现场检查所需材料
   - 整理合规证明文件
   - 提交监管报告

6. **合规培训**
   - 查看用户合规培训记录
   - 安排合规培训课程
   - 更新合规政策文档
   - 发布合规通知和提醒

#### 扩展场景
- **合规违规**：启动违规调查流程，记录和处理
- **监管检查**：配合监管机构现场检查
- **规则变更**：根据新法规更新合规规则

#### 后置条件
1. 合规检查完成，生成检查报告
2. 合规问题已记录和跟踪
3. 监管报告已准备或提交
4. 合规记录已归档

#### 业务规则
- 所有交易必须通过合规检查
- 合规检查必须有记录
- 合规问题必须及时上报
- 监管报告必须按时提交

#### 特殊需求
- 数据保留：交易记录必须保存6年以上
- 审计要求：审计日志必须完整不可篡改
- 监管合规：必须符合证监会和交易所要求

## 场景验证标准

每个场景应有对应的验收测试用例，验证标准包括：
1. **功能完整性**：场景所有步骤可正常执行
2. **性能要求**：满足场景的性能指标
3. **用户体验**：用户操作流程顺畅自然
4. **错误处理**：异常情况处理合理
5. **数据一致性**：场景执行前后数据状态正确
